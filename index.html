Avatar

Visual identifier for a person, product, or business entity. Displays a photo, initials, or placeholder/icon inside a circle (person) or square (product/object).

Anatomy

  1. 1. Avatar — main component (shape, color, content)
  2. 2. Border (optional) — subtle outline
  3. 3. Badge (optional) — interactive state indicator with icon

Dos and Don'ts

DosDon'ts
✅ Circle for people, square for products — always❌ No badges on avatars smaller than Small

✅ Interactive avatar → always tooltip + alt text❌ No unnecessarily large image files
✅ Badge → always tooltip
✅ Replace default product icon with a context-appropriate icon

Notes

  • color=Image does not appear in the standalone Avatar but shows up in Avatar children inside Avatar Group — different value in that context.
  • The border property is named optionalBorder inside Avatar Group children — same concept, different property name.
  • Badge None state (camera icon) is specifically for change-picture flows — not a generic "no badge" state.

Date/Time Picker

Combined input field + calendar/clock popover for selecting both a date and time in a single control. The Figma component shows the input field only — the popover (calendar + clock) is a separate nested structure revealed on button click.

Anatomy

  1. 1. Date/Time Picker input field — shows selected value; allows manual entry
  2. 2. Date/Time Picker button — trailing calendar icon; opens the popover
  3. 3. Date Picker container — calendar for selecting the date
  4. 4. Time Picker container — clock interface for hours, minutes, seconds
  5. 5. Footer — OK and Cancel actions

Dos and Don'ts

DosDon'ts
✅ Use Compact for desktop (L/XL), Cozy for mobile (S/M)❌ Do not show AM/PM toggle on 24-hour clock
✅ AM/PM Segmented Button for 12-hour clock only — never for 24-hour❌ Do not show week numbers in mixed calendar mode
✅ Always show a legend when special dates are displayed
✅ For timezone-sensitive input, offer a Select control alongside the picker
✅ Set min/max limits to restrict the selectable range
✅ Provide tooltips on the calendar icon button and the date/time sections

Notes

  • The Figma component node shows only the Input field — the calendar + clock popover is a separate nested structure that must be placed and connected independently in anatomy illustrations.
  • Calendar day cells can show up to 3 stacked outlines simultaneously (Current date + Selected + Focus) — this is correct behavior, not a design error.
  • The Compact/Cozy property is a density mode tied to device context, not a size property.
  • AM/PM Segmented Button is the one sub-element with strict conditional logic: 24-hour clock = never shown.

Time Picker

Input field + clock popover for selecting time only (hours, minutes, optional seconds). Supports 12-hour (AM/PM) and 24-hour formats. Like the Date/Time Picker, the Figma component shows only the input field — the clock popover is a separate nested structure.

Anatomy

  1. 1. Time picker input field — text field; accepts manual entry with or without mask
  2. 2. Time picker button — trailing clock icon; opens the popover
  3. 1. Time picker container — wraps the whole clock interface
  4. 2. Hours field — displays/edits the hour
  5. 3. Minutes field — displays/edits the minutes
  6. 4. Seconds field (optional) — displayed only when seconds precision is needed
  7. 5. AM/PM switch — Segmented Button for 12-hour format only
  8. 6. Clock face — circular selector; adapts to whichever unit is active (hours/minutes/seconds)

Dos and Don'ts

DosDon'ts
✅ Use Compact for desktop (L/XL), Cozy for mobile/tablet (S/M)❌ Do not show AM/PM toggle on 24-hour clock
✅ AM/PM switch for 12-hour format only — never for 24-hour❌ Do not use 24:00 end-of-day value with 12-hour/AM/PM format
✅ Hide seconds by default; show only when use case requires it
✅ Keep input mask on unless a locale specifically requires variable length
✅ Provide tooltips on the clock icon button and time unit fields
✅ For timezone-sensitive input, add a Select control next to the picker
✅ Set a default time to reduce user effort

Notes

  • The Figma component node shows only the Input field — the clock popover (Time Dropdown) is a separate nested structure with its own sub-elements.
  • The clock face circle does not receive keyboard focus — it mirrors the value in the fields but focus travels through Hours → Minutes → Seconds → Footer buttons only.
  • On mobile S, tapping the input field and tapping the clock button are two different interactions opening two different UIs (mini numeric popover vs full clock dialog).
  • Clock face sub-elements (Tick Marks, Number Selectors, Clock-face) are all base components marked "do not use directly" — they appear as anatomy callout targets but are not standalone insertable.

Date Picker

Input field + calendar popover for selecting a single date. Supports manual text entry and visual calendar-based selection. Like the other pickers, the Figma component shows only the input field — the calendar popover is a separate nested structure.

Anatomy

  1. 1. Date picker input field — text entry; accepts typed date or shows selected value
  2. 2. Date picker button — trailing calendar icon; opens the popover
  3. 3. Date picker popover — contains the full calendar
  4. 1. Calendar header — current month/year label; Previous (<) and Next (>) navigation buttons
  5. 2. Calendar grid — day cells arranged by week rows; week number column optional (left side)
  6. 3. Calendar day cell — individual date; can stack: Current date + Selected + Focus states (up to 3 outlines)
  7. 4. Legend (optional) — maps special date colors to meaning; up to 20 custom colors + standard colors for today/selected/working/non-working
  8. 5. Footer (optional) — OK and Cancel; only when explicit confirmation needed

Dos and Don'ts

DosDon'ts
✅ Use Compact for desktop (L/XL), Cozy for mobile/tablet (S/M)❌ Do not show week numbers in mixed calendar mode
✅ Match picker type to required granularity (day/month/year)❌ Do not allow year-range selection independently from year view
✅ Define min/max dates when business rules apply
✅ Always show a legend when special dates are displayed
✅ Provide tooltips on the calendar icon button and calendar
✅ Add Footer (OK/Cancel) only when explicit confirmation is required
✅ For timezone-sensitive input, offer a Select control alongside the picker

Notes

  • The Figma component is named Date (Range) Picker — the range behavior is a separate configuration of the same component, documented in the Date Range Selection entry.
  • The Footer is optional here — unlike Time Picker and Date/Time Picker where it always appears; Date Picker can close on selection without any footer. This is a genuine design decision per use case.
  • The Calendar component inside has its own properties (selection, mixedCalendar) that drive the picker type — these are not properties of the outer Date Picker itself.
  • The Header bar only exists on mobile S as a dialog title — it is absent from the desktop popover structure.

Popover

A small overlay window that opens on top of the current page to display additional information or actions in a compact way — without navigating away. Can contain any UI elements: fields, tables, images, charts, lists. Opened and closed by a trigger element (button, link, icon).

Anatomy

  1. 1. Header (optional) — title bar at the top; can be standard or custom
  2. 2. Subheader (optional) — secondary bar below the header; can contain any component; can replace the main header
  3. 3. Content area — main body; any component fits here; scrollbar appears automatically if content exceeds fixed height
  4. 4. Resize handle (optional) — bottom-right corner; appears when resizing is enabled
  5. 5. Footer (optional) — action bar at the bottom; OK (Emphasized) + Cancel (Transparent) buttons
  6. 6. Arrow (optional) — visual pointer connecting the popover to its trigger; can point left, right, top, or bottom

Dos and Don'ts

DosDon'ts
✅ Keep the trigger element visible while the popover is open❌ Do not use a header or footer when the content is self-explanatory and action-free
✅ Return focus to the trigger element when the popover closes❌ Do not position the popover so it obscures the trigger element
✅ Add a header when the content needs a title for context
✅ Add a footer only when the user needs to take an action
✅ Use modal behavior when the popover must stay open until explicitly dismissed
✅ Use navigation headers (back arrow) for multi-level flows inside a popover

Notes

  • The Popover has no component states of its own — states belong to the elements placed inside it, not the container. This is unique among all components documented so far.
  • Modal behavior is an invisible but critical property — it completely changes how dismissal works and is easy to miss in a design review.
  • The Subheader can replace the main Header entirely, or sit below it — two different use cases with different structural setups.
  • The content area is a free slot with no constraints — any component can go inside, including other popovers (for nested navigation flows).
  • The article contained leftover Input field guideline content from a template — this was excluded as it does not belong to the Popover.

Range Slider

A UI component that lets users select a value range (start + end) within a predefined numeric or non-numeric interval. Two independent handles define the minimum and maximum of the selected range. The track between them is the active (selected) portion.

Anatomy

  1. 1. Start point — minimum value of the scale (left endpoint marker)
  2. 2. Inactive track — unselected portions of the bar (outside the range)
  3. 3. Active track — selected range segment between handles
  4. 4. Left handle — sets the start value of the range
  5. 5. Right handle — sets the end value of the range
  6. 6. Value indicators (optional) — displayed above handles; can be read-only or editable input fields
  7. 7. Tick marks (optional) — dots/marks along the track at step intervals
  8. 8. Labels (optional) — text values at selected tick mark positions

Dos and Don'ts

DosDon'ts

Step Input

A numeric input with Decrease (−) and Increase (+) buttons on either side. Lets users adjust a numeric value in predefined increments or by typing directly. Always contains a value — never empty in regular state.

Anatomy

  1. 1. Subtract Button (−) — left icon button; decreases value by one step; disabled when minimum is reached
  2. 2. Value text — numeric value; right-aligned by default; editable when active
  3. 3. Add Button (+) — right icon button; increases value by one step; disabled when maximum is reached
  4. 4. Underline — bottom border line on the field container
  5. 5. Field container — wraps all three elements; field background + inner shadow

Dos and Don'ts

DosDon'ts
✅ Always set a meaningful step size for the use case❌ Do not leave the field in a truly empty state — default resets to 0
✅ Set min/max limits when a valid range applies❌ Do not use Step Input for non-numeric values
✅ Provide an initial value close to the expected typical input
✅ Use End alignment (default) for numeric values — easier to compare
✅ Use delayed validation when live feedback would be disruptive

Notes

  • The icons have their own dedicated interaction states (Hover icons, Active icons) — separate from the input field states. This two-axis state model (field vs. icons) is unique among all components in the database.
  • When max/min is reached, only the affected direction's icon is disabled — the other icon remains active. A common anatomy mistake is showing both icons disabled.
  • Value state messages appear below the field (not in a popover), and only while the field is active — different from Input where the message popover floats.
  • The Figma note about Clip Content is a known layout quirk — when the message popover should be visible in active state, clip content must be turned off on the component, otherwise the popover is hidden.
  • Step mode Multiple requires integer step sizes — it breaks with decimal steps. Worth knowing for configurations that mix decimal precision with step-mode logic.
